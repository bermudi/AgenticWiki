# Architectural Bloat

> A failure mode of automated Multi-Agent System (MAS) design in which the discovered architecture is structurally more complex than its functional behavior justifies. The architecture *looks* coordinated (multi-agent, role-specialized, debate-driven), but the actual reasoning path collapses to a simpler execution. The cost premium pays for structure that does not translate into functional utility — exposed by the [[multi-agent-illusion]] audit's architectural deconstruction.

## The Pattern

Architectural bloat is what you get when an automated search (MCTS, RL-trained orchestration, importance scoring, zero-shot coordination) optimizes a validation metric without a constraint on structural complexity. The optimizer finds the highest-scoring *configuration*; the configuration may include coordination machinery that is causally inert.

### Concrete Manifestations

**AFlow (code-based search, MCTS):** Of 14 final workflows generated by GPT-4o, GPT-5, and GPT-OSS-120B on five datasets, **7 (50%)** adopt the structure "iterate a single custom prompt three times, then aggregate" — functionally identical to standard CoT-SC. Four of these seven actually *underperform* the CoT-SC baseline: the extra orchestration machinery is a tax, not a benefit.

**ADAS (meta-agent + MCTS + validation archive):** Search dynamics on GPQA-Diamond (GPT-5) are non-monotonic. Accuracy frequently peaks early and subsequently regresses. The paper's interpretation: "performance gains are stochastic 'lucky' iterations rather than structural evolution." The archive accumulates the lucky iterations, not real architectural improvements.

**Motif analysis:** When the paper maps generated architectures to a rule-based dictionary (Self-Consistency, Aggregation, Verifier, etc.) across all settings, the primary positive signal originates from Self-Consistency motifs. On GPQA-Diamond, architectures with these motifs achieve 82.19% mean accuracy (+1.34pp over the global average). Specialized coordination motifs yield negligible additional gain — they are not *adding* value, they are *not subtracting* from the Self-Consistency baseline.

## What Causes It

Three mechanisms, all visible in the paper:

1. **The validation metric rewards structure, not function.** CoT-SC is a strong baseline. Anything that gets close to it is selected. The validation set is finite, so search converges on simple motifs that perform well on the validation set.
2. **MCTS exploration without a complexity penalty.** AFlow and ADAS explore the architecture space; the cost of an additional orchestration node is not in the objective. Adding nodes is free, so adding them is rational — even if their causal contribution is zero.
3. **The LLM-as-search-operator is itself a CoT-SC sampler.** The "meta-agent" in ADAS, the "controller" in MaAS, the "orchestrator" in MAS-Orchestra — these are LLMs. When they generate an architecture, they are doing a kind of inference. Their architectural choices inherit the same redundancy patterns as the workers.

## The Inverse: [[expert-mas]] Works Because It Avoids Bloat

The Expert-MAS baseline on [[smfr]] is hand-designed and *not* bloated: a Meta-Agent for parsing, an Extractor per investor for retrieval, a Calculator per investor for numerical reasoning, and a deterministic Python aggregator. Every component has a specific causal role. There is no "iterate a single custom prompt three times" because the hand designer chose not to include it.

The contrast is the diagnostic. Automated search discovers bloat; hand design avoids it.

## How to Detect Bloat

The paper proposes a mechanistic-interpretability criterion for MAS: **evaluate the degree to which assigned agentic roles exert measurable causal influence on the final decision.** Concrete tests:

- **Ablation cost test**: remove an agent or interaction edge; how much does accuracy drop? If removing an agent changes nothing, that agent is bloat.
- **Output influence test**: does the verifier or aggregator actually select outputs from the agents it claims to coordinate? If the verifier always picks the first worker's output (as in MAS-Zero), the rest of the pipeline is bloat.
- **Motif attribution**: decompose the discovered architecture into named motifs (Self-Consistency, Debate, Verifier, Aggregation); measure each motif's marginal contribution. If Self-Consistency explains all the gain, the rest is bloat.
- **Cost-quality Pareto position**: plot the architecture on cost vs. accuracy; if it sits on or below the CoT-SC Pareto frontier, it is bloat regardless of its absolute score.
