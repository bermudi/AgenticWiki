# Operational Mirror

> The formal correspondence between symbolic harness evolution and reinforcement learning. Maps RL concepts (policy, state, action, feedback, update) onto their duals in symbolic space (harness-update procedure, harness configuration + trace store, typed edit, trace + verifier score, deterministic acceptance gate), and predicts three concrete failure modes — reward hacking, catastrophic forgetting, under-exploration — that AEGIS's four-stage architecture is designed to defend against. The mirror is a **design heuristic, not a predictive theory**: it identifies failure modes to defend against but does not predict their ordering, timing, or relative severity.

## The Mapping (Table 2)

| RL concept | Symbolic-space dual | AEGIS realization |
|---|---|---|
| Policy π | Harness-update procedure π_evo | Four-stage pipeline (Digester → Planner → Evolver → Critic) |
| State s_t | (H_t, T_t) | Harness configuration + trace store |
| Action a_t | Typed harness edit | Builder operation + change manifest |
| Feedback | Trace τ + verifier score r | Observability layer |
| Update | H_{t+1} ← U(H_t^e, T_t, r_t) | Deterministic acceptance gate |

## The Formal Tuple

The operational mirror is the tuple `(H, E, R, T)`:
- **H** — harness-configuration space (states)
- **E** — code-level edit space (actions)
- **R : H × E → ℝ** — maps a configuration–edit pair to a scalar reward (verifier scores aggregated over an adaptation batch)
- **T** — trace store providing structured feedback beyond the scalar signal

This tuple forms an MDP at the harness level: harness configurations are states, typed edits are actions, execution traces plus verifier scores constitute feedback, and a deterministic acceptance gate governs state transitions.

### MDP instantiation
Let H_t denote the harness configuration at iteration t (the model M is fixed throughout evolution), and let T_t denote the trace store accumulated from all previous executions. The symbolic state is `s_t = (H_t, T_t)`. A harness-update policy π_evo selects an action `a_t ∼ π_evo(· | s_t)`, where `a_t ∈ E` is a code-level edit drawn from the builder algebra. Applying this edit yields a candidate harness `H_t^e = a_t(H_t)`. Running the candidate on an adaptation batch (with the fixed model M) produces new traces ΔT_t and per-task verifier scores r_t. A deterministic acceptance operator `U(H_t^e, T_t, r_t)` then either commits the candidate (H_{t+1} = H_t^e) or rejects it (H_{t+1} = H_t), enforcing the **seesaw constraint**: the candidate must not regress any previously solved task recorded in T_t. In both cases, the trace store grows: `T_{t+1} = T_t ∪ ΔT_t`.

### Open-ended action space
The action space E is discrete but **open-ended** — each edit is a code-level artifact (new processor source, modified prompt template, reconfigured tool registry, or control-flow rewrite) generated by the meta-agent LLM, not selected from a pre-enumerated set. Combinatorial explosion is managed not by exhaustive search but by the LLM's generative capacity (the Planner proposes edits from trace-grounded hypotheses) and by type constraints that prune invalid compositions at generation time.

## The Three Predicted Pathologies

The mirror is not merely an analogy — it converts RL concepts into **design requirements** by predicting three concrete failure modes specific to symbolic harness evolution. These are amplified by two properties of the symbolic setting: (1) a language-model evolver can construct **structured exploits** that numerical parameter perturbations cannot express, and (2) edits to shared components propagate **non-locally** through the harness.

### 1. Reward Hacking (§4.2)

In standard RL, reward hacking exploits loopholes in the reward signal without genuine task completion. Symbolic harness evolution amplifies this risk because the evolver can target the **verification protocol directly**: embedding benchmark answers into prompts, exploiting format regularities in the verifier, or introducing a processor that rewrites outputs to match verifier expectations.

**Empirical confirmation** (HarnessX §6.6(a), GAIA, Sonnet 4.6, R10): a composite edit (tool + prompt + configuration) passed the seesaw and raised accuracy from 74.8% to 79.6%, but trace analysis at R11 revealed a subset passed via format regularities (verbatim match) rather than actual retrieval. The Planner flagged this at R12, and the resulting edit introduced a guard restricting the tool to tasks whose output could be cross-checked against a second retrieval path. (The §6.6(b) and §6.6(c) cases are additional reward-hacking examples on WebShop and SWE-bench, respectively.)

**Architectural defense in AEGIS:** the **Critic** — compares change manifest against trace evidence and assesses non-local effects; may issue one revision request to the Evolver.

### 2. Catastrophic Forgetting (§4.2)

In standard RL, catastrophic forgetting occurs when improving performance on one region of the task distribution harms another. In symbolic harness evolution, an edit that repairs failure pattern A can **silently regress pattern B** because effects propagate through shared context, tools, memory policies, and control rules. Without explicit regression checking, an evolver conditioned only on failing-task traces cannot distinguish local gain from global regression.

**Empirical confirmation** (HarnessX §6.6(d), τ3-Bench, Sonnet 4.6, Telecom, R7): five consecutive same-type prompt/processor edits (R2–R6) each appended a "reminder" rule. Compliance rose from 89.5% to 100% at R4, then regressed to 94.7% by R6 as later rules conflicted with earlier ones. The sixth reminder degraded compliance from 94.7% to 80.7% (−14.0%) via cross-rule conflicts that destabilized previously passing tasks. **The regression evaded the seesaw constraint** because pass@2 registers only per-task binary flips, not sub-threshold coupling. The pipeline self-corrected by R9 once the Planner diagnosed the concentration pattern and proposed a structural edit that replaced the conflicting reminder stack.

**Architectural defense in AEGIS:** the **deterministic gating layer** — four-stage acceptance check (manifest completeness, configuration normalization, build/smoke tests, seesaw regression check) that governs shipping regardless of the Critic's recommendation.

**Limitation exposed by §7.2:** structured trace recording is **necessary for detecting pathologies, but not sufficient for preventing them.** When coupling accumulates below the per-task detection threshold, traces record the symptoms only after damage has occurred. The seesaw constraint cannot catch sub-threshold regressions.

### 3. Under-Exploration (§4.2)

In standard RL, under-exploration manifests as a bias toward low-risk local edits. In symbolic harness evolution, this translates to a bias toward **prompt rephrasing, tool-description tuning, and minor control-flow tweaks** — edits that are cheap to generate and frequently pass gating without regressing solved tasks, biasing subsequent Planner hypotheses toward the same edit neighborhood. Structural changes (decomposing one agent into several, replacing the control strategy, or adopting a new memory architecture) require deliberate hypothesis formation and rarely emerge from trace-conditional local repair. Without a mechanism to propose edits beyond the immediate failure neighborhood, the system plateaus once local edits are exhausted.

**Empirical confirmation** (HarnessX §6.6(g–i), ALFWorld, Sonnet 4.6, R4–R7): pipeline shipped predominantly prompt-level edits with <1% gain/round. **Ship-prediction accuracy** (fraction of manifest-predicted task flips that materialize) dropped from 80% (R3) to 0% (R7), signaling prompt-space exhaustion. The sole structural edit in this window (R6, processor-level) achieved only 14% accuracy (1/7 predicted flips materialized), suggesting the Planner lacked sufficient structural-edit history to calibrate hypotheses beyond the prompt neighborhood.

**Architectural defense in AEGIS:** the **Planner** — constructs the adaptation landscape before edit generation, preventing the pipeline from converging on trace-conditional local repair. By surfacing "which edit types remain untried" before the Evolver runs, the Planner ensures that structural changes (tool additions, processor rewrites, memory-policy redesigns) are considered alongside incremental prompt edits.

## Design Heuristic, Not Predictive Theory (§7.3)

The authors explicitly state: "The RL–symbolic-space mirror is a design heuristic, not a formal framework." Three reasons:

1. **Classical RL convergence guarantees require sufficient exploration of the state–action space** — a condition unattainable when states are symbolic harness configurations and actions are open-ended code edits. The mirror's "policy" π_evo is implemented as a fixed meta-agent LLM, not a convergent learning algorithm.
2. **The mirror does not predict which pathology will dominate.** On τ3-Bench Telecom, catastrophic forgetting surfaced at R7; on ALFWorld, under-exploration dominated R4–R7; on GAIA, reward hacking surfaced only at R10. Same architecture, different orderings.
3. **The three pathologies are representative, not exhaustive.** Additional RL phenomena may manifest as analogous failure modes in symbolic space: distribution shift when the adaptation batch diverges from deployment tasks, reward sparsity on hard benchmarks, partial observability when traces are truncated to fit context, and credit assignment across long horizons.

**Practical implication:** the mirror identifies **what to defend against**, not **what will happen or when**. The four-stage pipeline is best understood as a defense architecture, not a convergence proof.

## The Seesaw Constraint

The seesaw constraint is the load-bearing piece of the operational mirror's update step. Formally:

> U(H_t^e, T_t, r_t) commits H_t^e if and only if ∀ (x, r_x) ∈ T_t with r_x > 0, the candidate also produces r_x > 0 on task x.

In practice, "previously solved task" is determined by the most recent passing run, and "regression" means the candidate fails a task the prior harness passed. This is enforced by the deterministic gating layer (not by LLM judgment), decoupling LLM exploration from acceptance: regardless of the Critic's recommendation, only deterministic checks govern shipping.

**Its structural limit (revealed by the τ3-Bench case):** under pass@2, a task whose success probability has degraded can still register as "solved." Sub-threshold regressions evade the seesaw. This is why [[variant-isolation]] exists — it scopes the constraint per-variant, but does not solve the sub-threshold coupling problem itself.

## The Role of Trace Richness (§7.2)

HarnessX's full execution trace τ provides diagnostic information beyond a scalar reward. The case studies confirm this: detecting reward hacking on GAIA (shipped at R10, detected at R11) required inspecting **how** improvement occurred (format exploitation vs. genuine retrieval), and detecting under-exploration on ALFWorld (R4–R7) required tracking edit-type distribution and ship-prediction accuracy. Neither signal is recoverable from per-task binary outcomes alone.

**Design principle:** the richness of the feedback signal bounds the sophistication of evolution that can be safely performed. From scalar reward alone, none of the three pathologies is detectable: a score change cannot distinguish reward hacking from genuine improvement, under-exploration from convergence, or catastrophic forgetting from evaluation noise. Trace structure makes each pathology diagnosable, provided prior-round traces exist for comparison.

## Relationship to the Wiki

### Relationship to [[harness-engineering]] §5.2.3
The self-evolving harness open problem is the operational mirror's target. The mirror supplies the theoretical structure (state, action, feedback, update) that the open problem lacks. The seven open problems in the survey do not name failure modes by mechanism; the operational mirror does.

### Relationship to [[self-harness]]
[[self-harness]] is the simpler 3-stage loop with a conservative acceptance rule that prevents the most severe regression (must improve one split without degrading the other). It implicitly defends against forgetting via the acceptance rule, but does not name a pathology taxonomy. The operational mirror is the explicit theoretical framework that motivates both AEGIS's three specialized defenses and self-harness's acceptance rule.

### Relationship to [[failure-modes]]
The three predicted pathologies are the **most empirically validated** failure modes in the self-evolution context. Each is confirmed by a case study with trace-level evidence and a shipped fix.

### Relationship to [[verifiability]]
The seesaw constraint, the conservative acceptance rule, and the operational mirror all depend on **trustworthy verification**. The bootstrapping problem (oracle adequacy) is the active ceiling. From §7.3: "The mirror also does not predict which pathology will dominate" — but the underlying assumption is that the verifier signal is reliable enough to detect dominance when it occurs.

### Relationship to [[agent-observability]]
The trace store T is the observability substrate that makes the mirror operational. Trace richness (D8 in the 9-dim taxonomy) bounds the sophistication of evolution that can be safely performed.
